# 运行态/历史态 state 隔离与任务列表治理落地方案

## 目标

- 解决 temp_uploads 中运行态 state.json 持续产生导致历史任务列表污染问题
- 保留断点续传能力并提升历史任务列表可读性
- 保证 Go 服务端为唯一真相，前端只展示历史态

## 目录结构与文件约定

```
temp_uploads/
  runtime_states/   运行态状态文件
  history_states/   历史态状态文件
  task_xxx.txt
  task_xxx_translated.txt
```

状态文件命名：

```
<input_base_name>.state.json
例：task_123.txt.state.json
```

## 状态范围规则

| 状态                                     | 写入目录       |
| ---------------------------------------- | -------------- |
| queued / running                         | runtime_states |
| paused / interrupted / error / completed | history_states |

迁移规则：

- 进入 history 状态时删除对应 runtime 状态文件
- 进入 runtime 状态时删除对应 history 状态文件

## 服务端编码落地点（Go）

### 1. 路径与目录函数

新增常量与工具函数：

- `runtimeStatesDir` / `historyStatesDir`
- `ensureStateDirs()` 初始化目录
- `stateFileName(inputPath string)` 统一文件名
- `runtimeStatePath(inputPath string)` / `historyStatePath(inputPath string)`
- `statePathForStatus(inputPath, status string)` 根据状态选择目录
- `latestStateFile(dir, taskID string)` 读取最新状态文件

### 2. saveTaskState 写盘逻辑

- 写入路径按状态切换
- 同步删除另一目录中的旧文件
- deleted 状态不落盘

### 3. 读取逻辑

- `/api/task_status`：优先读 runtime_states，缺失再读 history_states
- `/api/tasks`：只扫描 history_states
- `/api/tasks/{id}`：优先读 runtime_states，缺失再读 history_states
- `/api/resume`：优先读 runtime_states，缺失再读 history_states

### 4. 删除逻辑

- `/api/tasks/{id} DELETE`
  - 如果任务运行中，先 cancel 并等待 DoneCh 或超时
  - 删除 runtime_states/history_states 中对应状态文件
  - 删除旧版本兼容路径 `temp_uploads/<input>.state.json`
  - 删除输入/输出文件

## 数据结构与状态字段

state.json 必含字段：

- id / status / total / current
- input_path / out_path
- instance_id / last_heartbeat_ts / status_reason
- src_file_name
- started_at / last_resume_at / elapsed_sec_accumulated

## 历史任务列表展示规则

- 历史任务只展示 history_states 中任务
- 文件名显示优先级：`src_file_name` > `basename(input_path)` > `Unknown Task {id}`
- 运行中任务只在“当前任务面板”可见

## 断点续传与 ETA 规则

- 断点恢复后 ETA 以累计耗时与当前进度计算
- 当 current=0 或 elapsed=0 时 ETA 返回 -1
- 前端显示“正在评估剩余时间...”
