# 增强任务管理及体验优化 PRD (历史任务删除、原始文件名记录与ETA修正)

## 一、 需求合并后的目标
- **历史任务支持删除**：删除不只是 UI 隐藏，必须清理落盘文件与内存状态。
- **state.json 增加 src_file_name**：真实记录用户上传时的原始文件名，并在历史列表中展示。
- **恢复后 ETA 准确**：断点恢复后 ETA 不可异常变小或抖动，应与当前进度一致，以历史速度为基准平滑延续。

---

## 二、 核心数据结构与状态转移机制

### 1. `state.json` (TaskState) 字段扩展
在原有的结构体上新增如下字段：
*   **`src_file_name`** (string): 用户上传时的原始文件名。若旧版本数据为空，前端 fallback 到 `input_file` 的 basename。
*   **`started_at`** (int64): 任务首次进入 running 对应的 Unix 时间戳。
*   **`last_resume_at`** (int64): 任务最近一次从 paused/queued 进入 running 的 Unix 时间戳。
*   **`elapsed_sec_accumulated`** (int64): 历史已沉淀的运行耗时（秒）。每次 Pause/Error 时累加。

### 2. 核心状态转移及时间计算约定
*   **Init -> Running**: 
    - `started_at` = Now()
    - `last_resume_at` = Now()
*   **Running -> Paused / Error**: 
    - `elapsed_sec_accumulated` += (Now() - `last_resume_at`)
*   **Paused / Error -> Running** (断点续传):
    - `last_resume_at` = Now() 
*   **Running -> Completed**:
    - `elapsed_sec_accumulated` = `elapsed_sec_accumulated` + (Now() - `last_resume_at`)

## 三、 API 接口清单与详细契约

### 1. [新增] 移除任务接口
* **Endpoint**: `DELETE /api/tasks/{task_id}`
* **逻辑说明**:
    1. 获取任务锁，判断状态机。
    2. 若状态为 `Running`，执行 `PauseSync()` 同步等待 Worker 彻底退出，防止异步 Cancel 中发生写回导致产生“幽灵文件”(脏写残留)。
    3. 从内存哈希表/队列摘除任务。
    4. 彻底删除物理文件（忽略不存在的错误）：`<task_id>.*`, `<task_id>_translated.*`, `<task_id>.*.state.json`。
* **返回格式**: `{ "task_id": "xxx", "status": "deleted" }`

### 2. [修订] 获取任务列表 / 获取详情接口
* **Endpoint**: `GET /api/tasks` & `GET /api/tasks/{task_id}`
* **逻辑说明**:
    * 返回数据中包含最新记录的 `src_file_name`。
    * 后端通过新公式推演当前真 ETA 以免导致除零异常：
      ```go
      current_elapsed := task.ElapsedSecAccumulated
      if task.Status == Running {
          current_elapsed += (time.Now().Unix() - task.LastResumeAt)
      }
      
      // 保护性计算除零风险
      if task.Current > 0 {
          speed := float64(current_elapsed) / float64(task.Current)
          task.EtaSeconds = int64(speed * float64(task.Total - task.Current))
      } else {
          task.EtaSeconds = -1 // -1 代表"计算中"或未知
      }
      ```

### 3. [修订] 提交任务接口
* **Endpoint**: `POST /api/upload` (或现有的上传接口)
* **逻辑说明**:
    * 从 `multipart/form-data` 中解析出 Header 里的原始文件名。
    * 强行写入 `TaskState` 的 `src_file_name` 字段中并落盘保存，满足“服务端单一真相”。

## 四、 前端交互设计与 UI 映射 (View Layer)

### 1. 列表文件名显示规范 (V-DOM 绑定)
* **主记录展示**：优先取 `task.src_file_name`，为空则 fallback 取 `task.input_path` 的 basename，如果都无效则显示 `Unknown Task {id}`。
* **副记录展示**：在文件名下方使用小号灰色字体展示系统生成的唯一 `task_id`，方便 Debug 和追溯。

### 2. 状态与 ETA 展现逻辑
* 若后端计算返回的 `eta_seconds == -1`，UI 应当转换为文案 `“正在评估剩余时间...”`，避免显示 `0` 秒以防给用户带来困扰或误导。
* 若断点恢复，因使用历史均速算法，进度条与倒计时应平稳延续。

### 3. 删除交互操作 (User Flow)
* **入口**：在历史任务表格列表的操作列加入一个明确的“垃圾桶”删除按钮 (使用警告色)。
* **防误删保护**：点击删除触发二次确认模态框（Modal）提示_“是否彻底删除该任务及关联文件？进行中的翻译将被中止并清除，此操作不可逆转。”_
* **响应**：确认后发起 DELETE 请求并附带 Loading 态。成功之后重刷/移除列表对应行。

## 五、 兼容性与边界处理补偿策略

1. **并发删除与文件IO冲突**
   * 必须实现带有一定同步性质的 Task Cancel（或通过“墓碑”状态标记机制拦截残余协程 IO 写回磁盘），从根本上保证真正无痕删除。
2. **前向兼容性处理**
   * 加载旧版未包含时间字段的 `state.json` 时进行平滑降级（如遇到读取旧的任务使得 `last_resume_at` 为 0 时可向前推取或容错忽略这部分统计表现），保障基础的进度展示稳定不受损。
